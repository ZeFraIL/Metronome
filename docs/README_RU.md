## 🎯 Цель проекта
*   **Какую задачу решает приложение:** Обеспечивает музыкантов надежным и точным метрономом для отработки ритма.
*   **Почему эта задача важна:** Ритм — это основа музыки. Точный метроном необходим как начинающим, так и профессиональным музыкантам.
*   **Целевая аудитория:** Музыканты, студенты музыкальных учебных заведений и преподаватели.

________________________________________
## 📌 Требования к приложению
### Функциональные требования
*   Воспроизведение звука "тик" с заданным BPM.
*   Визуальная анимация маятника, синхронизированная со звуком.
*   Регулировка BPM с помощью SeekBar.
*   Регулировка BPM путем перетаскивания грузика на виртуальном маятнике.
*   Функции Старт/Стоп.
*   Отображение названий музыкальных темпов (Grave, Allegro и т.д.).

### Нефункциональные требования
*   **Производительность:** Высокая точность тайминга для ударов метронома.
*   **Удобство использования:** Простой и интуитивно понятный интерфейс, имитирующий классический механический метроном.
*   **Надёжность:** Стабильная работа без отклонений от заданного темпа.

## 🧠 Общая архитектура
*   **Выбранный подход:**
    *   Простой MVC (Model-View-Controller).
*   **Почему выбран именно он:** Учитывая масштаб проекта, использование полнофункционального MVVM было бы избыточным. Текущая структура четко разделяет логику отрисовки (View) от логики управления (Controller).
*   **Основные компоненты системы:**
    *   `MainActivity`: Выполняет роль контроллера.
    *   `MetronomeView`: Кастомное представление (View).
    *   `SoundPool` и `ScheduledExecutorService`: Слой данных и логики для работы со звуком.

## 🧩 Подробное описание классов
### 📌 Класс: MainActivity
*   **Роль:** Точка входа и основной контроллер приложения.
*   **Ответственность:** Инициализация компонентов, обработка пользовательского ввода, управление жизненным циклом звука и анимации метронома.
*   **Основные методы:**
    *   `onCreate()` — Настройка UI и инициализация SoundPool.
    *   `startMetronome()` — Запуск планировщика звука и визуального аниматора.
    *   `stopMetronome()` — Остановка всех активных процессов.
    *   `calculateDuration()` — Логика преобразования BPM в миллисекунды.
*   **Взаимодействие с другими классами:** Взаимодействует с `MetronomeView` для обновления визуальной части и получает от него события изменения BPM.

### 📌 Класс: MetronomeView
*   **Роль:** Кастомный UI-компонент.
*   **Ответственность:** Отрисовка интерфейса механического метронома, маятника и шкалы темпов.
*   **Почему используется:** Для обеспечения уникального интерактивного опыта, имитирующего реальный физический прибор.

## 🔄 Диаграмма работы приложения (Сценарий)
1.  Пользователь открывает приложение.
2.  Пользователь настраивает BPM ползунком или перетаскиванием грузика маятника.
3.  Пользователь нажимает "Start".
4.  `ScheduledExecutorService` начинает воспроизводить звуки с точными интервалами.
5.  `ValueAnimator` двигает маятник синхронно со звуком.
6.  Пользователь нажимает "Stop" для прекращения работы.

## 🎨 UI/UX анализ
*   **Почему интерфейс сделан именно так:** Дизайн вдохновлен классическими механическими метрономами, что создает привычную рабочую среду для музыкантов.
*   **Какие принципы использованы:**
    *   **Простота:** Минимум кнопок, понятные элементы управления.
    *   **Логичность:** Физическая метафора (передвижение грузика) интуитивно понятна.
    *   **Доступность:** Крупный текст для отображения BPM.
*   **Что можно улучшить:** Добавление тем оформления (светлая/темная) или выбор различных звуковых семплов.

## ⚙️ Работа с потоками
*   **Используются:**
    *   `ScheduledExecutorService`: Используется для воспроизведения звука, чтобы обеспечить высокую точность независимо от нагрузки на UI.
    *   `ValueAnimator`: Работает в основном потоке для плавной отрисовки интерфейса.
*   **Почему выбран этот способ:** Обычные `Handler` или `Thread.sleep` могут давать погрешность. `ScheduledExecutorService` более надежен для ритмических задач.
*   **Как предотвращаются:**
    *   **Зависания (ANR):** Длительная логика воспроизведения вынесена из основного потока.
    *   **Утечки памяти:** Ресурсы освобождаются в методе `onDestroy()`.

________________________________________
## 💾 Работа с данными
*   **Где хранятся данные:** В текущей версии данные не сохраняются между сессиями.
*   **Почему выбран этот способ:** Для простого метронома достаточно настроек по умолчанию при каждом запуске.
*   **Как обеспечивается корректность:** BPM ограничен диапазоном от 40 до 208.

## 🐞 Обработка ошибок
*   Использование слушателя загрузки `SoundPool` гарантирует, что звук начнет воспроизводиться только после полной готовности.
*   Проверки на null для аниматора и планировщика перед их остановкой.

________________________________________
## ⚡ Производительность
*   **Оптимизации:** Использование `SoundPool` для минимизации задержек звука.
*   **Узкие места:** При очень высоких значениях BPM возможна нагрузка на UI-поток из-за частоты анимации, однако диапазон до 208 BPM является безопасным.
